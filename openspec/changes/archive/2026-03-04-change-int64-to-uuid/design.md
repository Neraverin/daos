## Context

The project currently uses `int64` for all ID fields (Host, Package, Deployment, Log). The company standard is to use UUID for all ID fields. This is a pre-1.0 change - no users or production data exist.

## Goals / Non-Goals

**Goals:**
- Replace all int64 ID fields with uuid.UUID across domain models, API, database, handlers, and TUI
- Use github.com/google/uuid for UUID generation
- Handle string → UUID conversion in handlers (inline, no helper functions)

**Non-Goals:**
- Migration script for existing data (not needed - no users yet)
- Database schema versioning/migration tooling (use simple new migration file)
- Helper functions for UUID parsing (inline uuid.Parse calls)

## Decisions

### 1. Use github.com/google/uuid library
- Already present in go.mod as transitive dependency
- `uuid.New()` generates random UUID v4
- `uuid.Parse()` parses string to UUID

### 2. SQLite stores UUID as TEXT
- SQLite has no native UUID type
- Store as TEXT, let Go handle conversion
- No sqlc type override - handle string ↔ UUID conversion in handlers

### 3. API uses string type for IDs
- OpenAPI: `type: string, format: uuid`
- Generated Go types: `string` (not uuid.UUID)
- Handlers parse string → uuid.UUID for internal use

### 4. Remove RETURNING clause from INSERT queries
- IDs are now provided by application, not generated by DB
- INSERT includes id as parameter
- RETURNING still used for other columns (created_at, updated_at)

### 5. Inline UUID parsing in handlers
- No helper functions
- Create: `uuid.New().String()`
- Get/Update/Delete: `uuid.Parse(ctx.Param("id"))`

## Risks / Trade-offs

- [Risk] UUID strings are longer than integers → slightly larger API responses
  - Mitigation: Acceptable tradeoff for company standard compliance

- [Risk] No foreign key enforcement on TEXT columns in SQLite
  - Mitigation: Existing foreign key constraints still work, just TEXT comparison

- [Risk] Path parameter parsing could fail with invalid UUID
  - Mitigation: Handler returns 400 Bad Request on parse error

## Migration Plan

1. Update OpenAPI spec (docs/openapi.yaml)
2. Run `make generate-api`
3. Create new migration file (002_uuid_schema.sql) with TEXT PRIMARY KEY
4. Update SQL queries to accept ID as parameter
5. Run `make generate-sql`
6. Update domain models (pkg/models/models.go)
7. Update handlers to generate/parse UUIDs
8. Update TUI models
9. Run tests: `make test && make build`
